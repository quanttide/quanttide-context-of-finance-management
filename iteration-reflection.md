# 迭代反思：简单的规则比复杂的制度更难设计

来自 2026-07-11 开发复盘。

## 原则

**简单的规则比复杂的制度更难设计。**

12 个字的决策机制（公开决议、记录账本、逐步补全）看似单薄，但够用。不够成熟和够用不是一回事。

## 代码层面的对照

| 反例 | 问题 | 应该怎么做 |
|---|---|---|
| `simulate` 先做交易层 | 需要手动调平，复杂不可维护 | 报表层算术，不碰交易记录 |
| `forecast` 周期性阈值拍脑袋定 | 同一个账户混合方向时漏检 | 先拿真实数据跑一遍再定规则 |
| `summary` 预留空命令 | 参数还没想清楚就占了位置 | 等有人提需求再做 |
| 均值+标准差做规律检测 | 数据稀疏时不可靠 | 更简单的方案：按账户+方向分开统计 |

## 信号

遇到"需要手动补偿"之类的复杂实现时，停下来问：**是不是抽象层次选错了**。

复杂度陡增往往是设计错误的信号，不是"这个功能本来就这么复杂"。

## 工作流

1. 先写一行注释描述"最简有用形式"
2. 如果这行注释看起来太复杂，说明方案还不够简
3. 拿真实数据验证假设，再写代码

## 解决问题 vs 构建系统

回头看今天做的事，区别很清楚：

| 做事方式 | 解决问题 | 构建系统 |
|---|---|---|
| 对象 | 一个具体问题 | 一类问题的结构 |
| 成果 | bug fix / 一次回答 | 规则、流程、CLI 命令 |
| 寿命 | 用完即止 | 可以反复用 |
| 进化方式 | 下一个问题再修一次 | 系统容量不够时升级结构 |

### 今天发生的系统升级

- `principles.md` → `procurement.md`（合并：原则和案例不该拆）
- `procedure.md` → handbook（移动：流程是方法不是政策）
- `rules.md` → `rules/`（展开：一个文件装不下时自然变目录）
- `troubles/` → handbook（提升：问题描述是通用知识不是团队隐私）

每次都不是"改内容"，而是"升级结构"。问题不是被解决，是结构进化后问题自然不存在了。

### 还没建完的系统

- **产品流水线**：journal→profile→handbook→CLI 已跑通但还没对外交付
- **决策工具化**：周会决议→账本记录是流程但还不是工具
- **教训传感器**：`iteration-reflection.md` 记录错误信号，下次遇到就知道要换抽象层次
